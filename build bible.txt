AECO TRUTH ENGINE — BUILD BIBLE (v1.0)
Offline, DB-First, Strict-Ground-Truth Assistant
Target: Windows desktop app + local backend + local DB + local LLM

=====================================================================
0) TL;DR FOR THE TEAM
=====================================================================
We are NOT building “chat with files”.
We are building an OFFLINE ENGINEERING TRUTH SYSTEM:

- Ingest engineering artifacts (IFC/BIM, P6, PDFs, Office, emails, logs).
- Run deterministic extractors -> structured staging tables with provenance.
- Run deterministic fact builders -> explicit computed facts with lineage.
- Validate/certify facts via rules + human queue.
- LLM is a query planner + narrator. It can answer ONLY from CERTIFIED facts.
- If facts are missing, LLM must refuse and output a coverage gap + job plan.

Non-negotiable: If it’s not in certified facts with citations, we do not answer.

=====================================================================
1) DEFINITIONS (STRICT SEMANTICS)
=====================================================================
1. Evidence
   Atomic extracted item tied to a specific file version and location.
   Examples: OCR span with bbox, PDF text block, Excel cell, P6 activity record,
             IFC property, email header field.

2. Extraction
   Structured output of an extractor: a set of evidence records + metadata + diagnostics.
   Always reproducible from the file version + extractor version.

3. Canonical Entity
   Normalized “real project object” across sources.
   Examples: Drawing (by drawing number), Document revision, Activity, IFC element GUID,
             Equipment tag, RFI, Submittal, Permit.

4. Crosswalk Link
   Explicit link between canonical entities.
   Examples: Activity -> Zone, Activity -> Elements, RFI -> Elements, Submittal -> Equipment.
   Links can be VALIDATED or CANDIDATE; assistant can only rely on VALIDATED links.

5. Fact (Ground Truth Unit)
   A computed/validated statement allowed to be used in an answer.
   Must include: fact_type, subject_id, scope, as_of snapshot, typed value, method version,
                status, and lineage to evidence + links.

6. Snapshot (As-Of State)
   A coherent “project state” that answers refer to:
   - schedule snapshot (data date, baseline)
   - design snapshot (model revision/issue package)
   - docs snapshot (latest revisions set)
   - registers snapshot (RFI/submittal/procurement/field logs cut-off)

7. Certified Facts
   Facts with status=VALIDATED (automated rule pass) and if required, HUMAN_CERTIFIED.

=====================================================================
2) PRODUCT GUARANTEES (ACCEPTANCE CRITERIA)
=====================================================================
G1. Strictness:
    The assistant must refuse any claim that cannot be fully supported by certified facts.

G2. Reproducibility:
    Given same file versions + extractor/builder versions, the DB can be rebuilt and match.

G3. Provenance:
    Every answer must cite facts, and every fact must cite evidence locations.

G4. Version correctness:
    Facts are computed “as-of” an explicit snapshot; superseded facts remain in history.

G5. Conflict safety:
    If conflicting certified facts exist, assistant must disclose conflict; no silent choice.

=====================================================================
5) PIPELINE + JOB SYSTEM
=====================================================================
5.1 Jobs
Each job is idempotent and versioned. Jobs form a dependency DAG.

Job types:
- INGEST_FILE_VERSION(file_path)
- EXTRACT(file_version_id, extractor_id)
- NORMALIZE(project_id, file_version_id)
- BUILD_FACTS(project_id, builder_id, snapshot_id)
- VALIDATE_FACTS(project_id, fact_type or builder_id)
- RECONCILE_CONFLICTS(project_id)
- UPDATE_SNAPSHOT(project_id)

5.2 Dependency Graph (high level)
INGEST -> EXTRACT -> NORMALIZE -> BUILD_FACTS -> VALIDATE -> CERTIFY -> SNAPSHOT_UPDATE

5.3 Incremental processing
On file change:
- create new file_version
- run relevant extractors
- recompute only dependent canonical entities and fact builders
- supersede prior facts for the same subject/as-of rule

=====================================================================
6) EXTRACTOR SDK (PLUGIN CONTRACT)
=====================================================================
6.1 Required interface
Extractor:
- id: string
- version: semver
- supported_file_types: list
- run(file_version_id) -> ExtractionResult

ExtractionResult must include:
- records: list of typed staging rows OR direct inserts via repository layer
- diagnostics: warnings/errors
- metrics: runtime, counts, confidence stats
- provenance: always embedded per record (file_version_id + location)

6.2 Provenance requirements by file type
- PDF: page_no + bbox + (render_cache_path optional)
- Excel: sheet + cell_ref
- Word/PPT: block index + slide no
- P6: activity_id/wbs_id keys
- IFC: ifc_guid + pset/prop names
- Email: message_id + header fields + attachment mapping
- Registers: row number + source file reference

=====================================================================
7) FACT BUILDER SDK (DETERMINISTIC ENGINES)
=====================================================================
7.1 Required interface
Builder:
- id: string
- version: semver
- required_inputs: staging tables + canonical entities + link types
- run(project_id, snapshot_id) -> list[fact] + list[fact_input] + diagnostics

7.2 Builder must be deterministic
No LLM in builders for producing validated facts.
If ML/LLM is used, output must be CANDIDATE and go to validation queue.

=====================================================================
8) VALIDATION + CERTIFICATION
=====================================================================
8.1 Automated validation (must exist before “validated”)
Rules (examples):
Schedule:
- no relationship cycles
- no orphan activities on critical path
- finish >= start
- data date within project range
BIM:
- GUID uniqueness
- spatial containment completeness
- required properties present (per discipline)
Docs:
- latest revision selection deterministic
- supersession chain consistent
Crosswalk:
- ACTIVITY_TO_ELEMENTS links must have evidence or be human-validated
- one-to-many constraints (configurable)
Procurement:
- dims/weights parseable and units normalized

8.2 Human certification queue (mandatory for these default categories)
- OCR-derived title block critical fields when confidence < threshold
- inferred Activity<->Element links (unless imported from 4D tool)
- extracted action items/commitments from emails
- any “judgment-based” compliance statement

8.3 Fact usability policy
Assistant may only use:
- status=VALIDATED
- plus required certification flag if rule group demands it

=====================================================================
9) SNAPSHOT SYSTEM (“AS-OF TRUTH”)
=====================================================================
9.1 Snapshot types
- SCHEDULE_SNAPSHOT: derived from a specific P6 import, includes data_date and baseline tag
- BIM_SNAPSHOT: derived from a specific IFC/model issue package
- DOC_SNAPSHOT: derived from document_revision set (latest or issue package)
- REGISTER_SNAPSHOT: derived from latest ingested logs at a cut-off time
- PROJECT_SNAPSHOT: references the above for a coherent as-of state

9.2 Snapshot selection
- “latest certified snapshot” is default for chat
- user can ask “as of issue package X” or “as of data date Y”

=====================================================================
10) FACT QUERY API (WHITELISTED, NOT FREE SQL)
=====================================================================
LLM does NOT get raw DB access. It gets a strict set of tools returning Fact Packets.

10.1 Tools
- coverage_check(question_template_id, project_snapshot_id, parameters_json) -> CoverageReport
- fact_get(fact_type, subject_kind, subject_id, asof_json) -> FactPacket
- fact_list(fact_type, filters_json, asof_json, limit, sort) -> FactPacket
- link_get(link_type, from_kind/from_id filters, to_kind filters, status=VALIDATED) -> LinkPacket
- entity_get(entity_kind, filters_json) -> EntityPacket
- conflict_check(scope_json, asof_json) -> ConflictPacket

10.2 FactPacket response contract
- facts: list of fact rows
- lineage: fact_inputs for each fact
- snapshot: echoed snapshot/as-of
- completeness: complete/partial
- conflicts: any conflict references
- notes: warnings for UI

=====================================================================
11) LLM ORCHESTRATION (STRICT ANSWERING PROTOCOL)
=====================================================================
LLM steps (hard enforced):
1) Determine question template ID + parameters.
2) coverage_check(template, snapshot, params).
3) If coverage incomplete -> REFUSE:
   - “Not available in DB”
   - list missing fact_types and missing link_types
   - propose pipeline jobs / required imports
4) If coverage complete:
   - fetch facts/links only via APIs
   - answer with citations:
     - cite fact_id(s)
     - include evidence refs (page/cell/GUID/activity keys)
5) If conflict exists:
   - disclose conflict and show both sets (or refuse if policy requires)

LLM output formats:
- “Answer” section
- “Evidence” section (fact IDs + source pointers)
- “Assumptions” section must be empty unless user explicitly allows assumptions (default: no assumptions)

14) HOW WE ANSWER THE 200 QUESTIONS (MECHANICALLY)
=====================================================================
Each user question:
- maps to a template (T01..T40)
- requires a set of fact types + link types
- coverage_check determines if we can answer
- if yes -> fetch facts -> render answer + citations
- if no -> refuse + list missing items + propose pipeline jobs

NO OTHER PATH EXISTS.

16) TESTING / QA (NON-NEGOTIABLE)
=====================================================================
Strictness tests:
- If a required fact is missing -> assistant must refuse
- If fact exists but status != VALIDATED -> must refuse
- If link exists but status != VALIDATED -> must refuse
Provenance tests:
- Every fact must have at least 1 fact_input
- UI must open evidence location (page/cell/GUID)
Rebuild tests:
- Wipe DB -> rebuild -> same facts count + same values for deterministic builders
Conflict tests:
- Create conflicting doc latest revision -> assistant must disclose conflict
Performance tests:
- Incremental update only recomputes dependent builders
