Mental model of the product
What the user is really doing

They’re not “chatting with files.” They’re building and maintaining an offline project truth database:

Ingest artifacts → immutable versioned registry

Extract deterministic evidence → typed staging tables with provenance

Normalize / Link entities across sources → validated crosswalk links

Build facts deterministically → fact rows + lineage to evidence

Validate / Certify facts → rules + human queue gating “answerability”

Query via Chat → LLM is only a planner/narrator, never a source of truth

BPMN-style step list with lanes
Lane: User

Create/open a project.

Import one or more project artifacts (P6 XER/XML, IFC, PDFs, Office, registers).

Monitor processing; resolve ingest/extract errors if any.

Review generated facts and any candidate links/facts that require approval.

Certify/approve required items in the validation queue.

Ask questions in Chat (optionally specifying “as-of” snapshot).

Export fact packs / reports as needed.

Lane: UI (Dashboard/Documents/Facts/Schedule/Chat)

Project selection: Open existing project or create a new one.

Documents page: Add files (drag/drop, folder import, or wizard).

Dashboard page: Show job queue status, throughput, failures, pending validations, coverage status.

Facts page: Show coverage dashboard + fact explorer + conflicts.

Schedule page: Show schedule snapshot selection + schedule visualizations and health checks.

Chat page: Enforce strict answering protocol and show Evidence/Sources/Activity log.

(Those exact modules exist in your tree: src/ui/pages/dashboard_page.py, documents_page.py, facts_page.py, schedule_page.py, chat_page.py.)

Lane: Job System (job queue + manager + jobs)

Enqueue INGEST job(s) per imported file.

After ingest, enqueue EXTRACT job(s) for relevant extractor(s) by file type.

After extract, enqueue NORMALIZE / LINK job(s) to create canonical entities + candidate/validated links.

After normalize/link, enqueue BUILD_FACTS jobs per builder + snapshot.

After build, enqueue VALIDATE_FACTS and RECONCILE_CONFLICTS, then UPDATE_SNAPSHOT.

Persist job outcomes, diagnostics, and any “requires human certification” items.

This is explicitly the intended dependency DAG:
INGEST → EXTRACT → NORMALIZE → BUILD_FACTS → VALIDATE → CERTIFY → SNAPSHOT_UPDATE.
